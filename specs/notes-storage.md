# Notes Storage

> Status: **Aspirational**
> Last updated: 2026-04-27

How non-markdown attachments are stored, served, and addressed. The store is **Garage**, an open-source S3-compatible object store. See [`notes-data-model.md`](notes-data-model.md) for the metadata table.

## Why Garage

- Open source and self-hostable; replaces Minio (which moved away from FOSS in 2025).
- Single static binary, runs in docker-compose with one persistent volume.
- S3 API compatibility lets us use any AWS SDK or `aws-sdk-rust` in the API service.
- Replicates well across nodes if we ever scale beyond one VPS (not relevant yet).

Cloudflare R2 was considered as the alternative; rejected because the operator prefers self-hosted on this VPS, has 100 GB available, and wants to keep notes-api dependencies inside the Tailnet. R2 stays as a "see also" if Garage maintenance becomes a problem.

## Bucket layout

One bucket: **`notes`**.

```
notes/
└── attachments/
    └── {sha256}/
        └── {filename}
```

Examples:
- `attachments/9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08/diagram.png`
- `attachments/9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08/diagram.png` ← same blob, different note paths share the object

The hash directory makes the URL **content-addressed**: changing the file produces a new URL, so caches never go stale. The trailing filename is purely cosmetic — it lets browsers display a sensible "Save As..." name and surfaces the file extension to MIME inference. The Postgres `attachments_metadata.path` column carries the vault-relative path, decoupled from the object key.

## Upload flow

1. Plugin notices a new/changed non-markdown file.
2. Plugin computes SHA256 client-side.
3. Plugin sends `PUT /api/v1/sync/attachments` with `multipart/form-data` (metadata + binary).
4. API:
   - Verifies SHA256 matches its own computation.
   - If `attachments_metadata` already has a row at this `path` with the same hash → no-op (200 with current revision).
   - If hash is new but Garage already has the object (some other path uploaded the same content earlier) → skip the S3 PUT, just upsert the metadata row.
   - Else PUT to Garage at `attachments/{sha256}/{filename}` with appropriate `Content-Type`.
   - Bump revision, append to `change_log`, return.

The plugin never holds Garage credentials. The API is the only writer.

## Download flow

Two patterns, both supported:

- **Public published** (used by the blog): `GET /api/v1/notes/:type/:slug` returns body with image references already rewritten to **direct Garage URLs**. The blog's HTML loads images from `https://notes.tail-XXXXX.ts.net/attach/{hash}/{filename}` — Caddy proxies `/attach/*` to Garage with `Cache-Control: public, max-age=31536000, immutable`. Content-addressing makes "immutable" safe.
- **Admin / drafts**: same URL space. Anything in the bucket is reachable by anyone on the Tailnet who can guess the SHA256 — practically unguessable (256 bits of entropy), but if we ever need stronger isolation for embargoed content, switch to **time-bounded signed URLs** generated by the API.

## Markdown rewriting

Obsidian authoring syntax is preserved in the database. Resolution happens at render time, in the API's `GET /v1/notes/...` handlers (so the blog and admin both get identical resolved bodies):

- `![[image.png]]` → look up `attachments_metadata WHERE filename = 'image.png' AND deleted_at IS NULL` → rewrite to `![](https://.../attach/{hash}/image.png)`.
- `![[Folder/sub/diagram.svg]]` → exact-path match preferred, filename fallback.
- Multiple matches by filename: prefer the closest-by-path attachment to the note's path; tie-break by most recent `modified_at`. If still ambiguous, leave the link broken and add a render-time warning visible in the admin's preview view.

Wikilinks for *notes* (`[[Other Note]]`) follow the rules in [`notes.md`](notes.md) — handled separately.

## Lifecycle and GC

- **Soft delete only** at the metadata level. The API never directly deletes Garage objects in response to a `DELETE /v1/sync/attachments`.
- A nightly GC job in the API:
  1. Find `attachments_metadata` rows with `deleted_at < NOW() - INTERVAL '90 days'`.
  2. For each, check if any other live row references the same `content_hash`.
  3. If not, hard-delete the metadata row **and** the Garage object.
- This protects against accidental delete: a 90-day window is long enough to notice and undelete via the admin UI.
- Garage replication is a deploy-time concern; for one VPS, replication factor is `1`.

## Local development

The notes-api docker-compose includes Garage configured for single-node mode:

```yaml
garage:
  image: dxflrs/garage:v1.0.1
  volumes:
    - garage-data:/var/lib/garage/data
    - garage-meta:/var/lib/garage/meta
    - ./garage.toml:/etc/garage.toml:ro
  ports:
    - "127.0.0.1:3900:3900"   # S3 API, bound to localhost only
    - "127.0.0.1:3902:3902"   # admin API
```

`garage.toml` uses `replication_mode = "none"` for dev. First-run bootstrap (creating the layout, the bucket, and an access key) is captured in a `notes/scripts/bootstrap-garage.sh` so it's reproducible — see [`notes-api-deployment.md`](notes-api-deployment.md).

## Limits

- **Per-attachment cap**: 25 MB (config in API env, surfaced to plugin via `/v1/health`). Larger files are rejected with `413` and the plugin logs + skips.
- **Total bucket size**: 100 GB VPS minus Postgres + system overhead → conservatively budget **70 GB** for Garage. The admin dashboard surfaces current bucket usage so we notice before we hit the cliff.
- **Per-extension allow-list**: not enforced. We trust the operator's vault content.

## What's not in v1

- Image transformations (resize, format conversion). The blog can ship `srcset` once we add a Sharp-based proxy or move to R2 + Cloudflare Images. Punt.
- Cross-region replication.
- Public bucket policies (everything is gated by Caddy / Tailscale today).
